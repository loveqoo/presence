# 리팩토링 규칙

## 복잡도 기반 리팩토링

리팩토링 전 복잡도를 측정하고 데이터 기반으로 우선순위를 정한다. 복잡도가 높은 파일부터 착수한다.

| 유형 | 전략 |
|------|------|
| 긴 조건 체인 | 맵/테이블 디스패치, 다형성 |
| 깊은 중첩 | 조기 반환, 함수 추출 |
| 복합 조건 | 술어 함수 추출 |
| 긴 함수 | 단일 책임 단위로 분리 |
| 콜백 중첩 | 파이프라인 합성, 모나드 체이닝 |
| 같은 조건 반복 | 한 곳으로 응집 (전략 패턴, ADT 매칭) |

## 구조 우선 원칙

구현의 효율성보다 구조가 주는 효율성을 우선한다. 지금 비어 있는 껍데기라도, 전체를 일관되게 형성하는 구조라면 유지한다. 같은 레벨의 관심사가 서로 다른 형태를 가지면 소비처에 분기가 생긴다.

## 구조 가시성 원칙

내부에 구조(포함/소유 관계)가 존재한다면 코드에 명확하게 드러나도록 한다. 또한 그 구조의 의존 관계도 명확해야 한다.

- 포함 관계는 필드로 드러낸다: `parent.child` 형태로 접근 가능해야 함. 형제 필드로 나란히 두고 외부에서 수동 조립하지 말 것.
- 의존 방향은 생성 순서·소유권으로 드러낸다: 소유자가 구성원을 만들고 수명을 관리. 구성원이 소유자를 파라미터로 받아 다시 돌아오는 역방향 금지.
- 같은 수명을 가진 객체들은 하나의 컨테이너로 묶는다. 생명주기가 다른 것들만 형제로 둔다.
- 소비처에서 생명주기·조립 순서를 수동 조율하게 만들지 말 것 — 이는 구조가 숨겨졌다는 신호.

## 타입 분기 → 다형성

타입 플래그로 분기하는 코드가 보이면 서브클래스로 전환한다. 공통 클래스에서 알고리즘을 정의하고, 유형별 서브클래스에서 각 단계를 구현한다.

## 매직 스트링 / 매직 넘버 금지

도메인 의미를 가진 리터럴 값은 반드시 `Object.freeze({...})` enum 상수로 추출한다.

- **금지**: `x === 'local'`, `status === 'completed'`, `timeout === 30000`, `type === 'admin'` 등 의미 있는 리터럴 직접 비교/할당
- **허용**: 진짜 일회성 값(에러 메시지, 로그 텍스트, 단일 문맥 suffix), wire format에 명시된 프로토콜 상수 정의부 자체, 테스트 계약 검증 assertion
- **상수 값은 문자열 유지 가능** — 직렬화/trace/wire 호환을 위해 `Object.freeze({ LOCAL: 'local' })`처럼 값은 string, 참조만 상수로 통일
- **배치**: 같은 도메인에서 공유되면 도메인 모듈에 (예: `DelegationStatus` in `delegation.js`). 여러 도메인 횡단이면 `policies.js`에
- **정의와 사용은 한 enum을 공유** — getter 리턴값, 기본 파라미터, 비교문, 디스패치 조건 모두 같은 상수를 참조해야 함

금지 신호: 같은 문자열/숫자 리터럴이 두 파일 이상에 등장하면 즉시 enum으로 추출.

## Extract Class 판단 기준

다음 신호가 보이면 Extract Class를 적용한다.

| 신호 (Code Smell) | 판단 기준 | 조치 |
|---|---|---|
| **Long Parameter List** | 함수 파라미터 5개 초과 (`eslint max-params: 5`) | (a) Reader DI 환경 객체로 묶음, (b) 옵션 객체 destructuring `({ a, b, c, d, e, f })`, (c) 공유 설정을 인스턴스 필드로 |
| **Data Clump** | 같은 묶음이 반복 (파라미터, 맵 키셋, 설정 그룹 등 형태 무관) | 묶음을 클래스로 응집 |
| **Pass-through** | 함수 A가 받은 인자를 거의 그대로 함수 B에 전달 | A와 B를 같은 클래스의 메서드로 |
| **Feature Envy** | 여러 함수가 같은 상태의 필드를 반복 참조 (독립 함수, 맵 값, 콜백 등 형태 무관) | 그 상태를 가진 클래스의 메서드로 이동 |

## Extract Class 적용 원칙

- 클래스는 설정 응집 + 메서드 그룹화 용도. 상속 계층은 최소화
- 순수 로직은 메서드 안에서도 FP 규칙 준수 — Either, Free, Writer 등 모나드 사용 유지
- constructor에서 설정, 메서드에서 행위 — 설정 pass-through 제거
- 이중 패턴(`x`/`getX`)은 통일 — `resolveX` 하나로 수렴

## React/Ink 컴포넌트 리팩토링

컴포넌트가 커지면 로직을 custom hook으로 추출하여 컴포넌트는 렌더 트리 조립에만 집중시킨다.

| 신호 | 추출 대상 |
|------|-----------|
| 여러 useEffect가 같은 관심사를 처리 | 하나의 custom hook으로 통합 (예: `useAgentMessages`) |
| 핸들러가 거대한 컨텍스트 객체를 조립하여 전달 | 컨텍스트 조립 + 디스패치를 custom hook으로 캡슐화 |
| 관련 없는 상태가 한 컴포넌트에 혼재 | 관심사별 custom hook으로 분리 |

적용 원칙:
- custom hook은 `hooks/` 디렉토리에 배치, `use` 접두사로 네이밍
- hook은 상태 + 부수효과를 캡슐화하고, 컴포넌트에 최소한의 인터페이스만 노출
- 본질적 UI 복잡도(조건부 렌더링, props 분기)는 컴포넌트의 자연스러운 책임이므로 무리하게 추출하지 않는다
- 렌더러 디스패치 테이블, 구문 하이라이터 등 순수 매핑 로직은 구조가 이미 깨끗하면 그대로 둔다

## 모듈 스코프 최소화

모듈 스코프의 독립 함수는 최소화한다. 클래스 인스턴스를 통해서만 사용되는 함수는 해당 클래스의 메서드로 흡수한다.

## 클래스 분리 판단 기준

한 클래스가 독립적으로 변경될 수 있는 여러 책임을 가지면, 책임별로 별도 클래스로 분리하고 조율 객체가 조합한다.

## 네이밍

- 언더바 접두사(`_name`) 금지 — 함수/메서드 내부의 일시적 변수는 예외
- 함수/메서드 이름은 구현 내용을 충분히 반영해야 한다
- 파라미터 순서: 반복 사용되거나 필수인 파라미터를 앞쪽에 배치
- 파라미터가 많으면 필수와 옵션을 구분하여 격리한다 (필수는 positional, 선택은 옵션 객체)
- 대문자: 불변 값 (열거 태그, 상수), 파스칼케이스: 생성자 (클래스, 팩토리), 카멜케이스: 함수·변수·인스턴스
- 개발자만 보는 에러 메시지는 영어로 작성한다
- 한 글자 변수명(`s`, `e`, `x` 등) 금지 — 의미를 드러내는 이름을 사용한다

## 파일명

- 기본은 kebab-case로 통일한다
- 외부 생태계의 관례가 강제되는 경우만 예외로 허용하며, 이 때 파일명은 주 export 식별자의 표기법과 일치시킨다

## 모듈 분리 판단 기준

- 한 파일에 관심사가 다른 함수/상수가 나열되면 모듈 분리
- 정책 상수는 `policies.js`에 통합. 파일별 로컬 상수 금지
- 유틸리티 함수가 `fun-fp.js`에 이미 존재하면 재정의하지 않고 import
- re-export로 창구 역할을 하는 파일 금지 — 소비처가 원천 모듈에서 직접 import
- import한 객체를 re-export 금지 — 모듈은 자신이 정의한 것만 export
- 사용하지 않는 import 객체 제거
- 사용하지 않는 함수/메서드는 제거하거나 사용하는 곳으로 옮긴다
- 테스트에서만 사용되는 함수/메서드는, 다른 public API로 동일한 검증이 가능하면 제거한다
- import 구문 사이에 코드를 넣지 않는다
- 필요 없는 기능은 제거한다
- 한 곳에서만 사용되는 함수는 사용처에 인라인한다
- 상태 분기(switch case, 메시지 핸들러 등)에는 "이 상태에서 무엇을 하는가"를 한 줄 주석으로 명시한다
