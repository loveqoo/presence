# 빈 워크스페이스에서 외부 파일 요청 시 LLM 응답이 도움이 되지 않음

**영역**: TUI
**심각도**: medium
**상태**: open
**관련 코드**: `packages/infra/src/infra/local-tools.js`, `packages/infra/src/i18n/ko.json` (error.file_not_found)

(REGISTRY: FP-69)

## 시나리오

신규 가입 직후 빈 워크스페이스(`~/.presence/users/{username}/`) 상태에서 TUI를 시작한 유저가 "package.json 파일 읽어줘"라고 입력한다. 해당 파일은 유저의 워크스페이스가 아닌 개발 프로젝트 디렉토리에 존재한다.

## 현재 동작

1. W1 workingDir 단일 규칙으로 에이전트의 파일 작업 기준점은 `~/.presence/users/{username}/`이다.
2. LLM이 `read_file("package.json")`을 시도 → `error.file_not_found: "파일을 찾을 수 없습니다: package.json"` 반환.
3. LLM이 재시도(retry) → timeout(120초).
4. 유저는 결국 "파일을 찾을 수 없습니다"만 반복해서 받는다.

라이브 e2e 관찰(과제 기술 참조): LLM이 찾지 못하는 이유를 유저가 이해할 수 있는 방식으로 안내받지 못해 최대 120초 대기 후 실패.

## 마찰 포인트와 FP-65/66 과의 관계

FP-65(resolved)는 "경로 차단 에러 메시지가 존재하지 않는 allowedDirs 설정을 안내"하던 문제였다. 현재는 `error.access_denied`가 "워크스페이스 밖입니다. 워크스페이스 내부 경로로 요청하세요"로 올바르게 표시된다.

FP-66(resolved)은 StatusBar dir 세그먼트가 TUI cwd 대신 워크스페이스를 표시하도록 수정되었다.

**이 FP-69는 다른 패턴이다.** 문제는 에러 메시지의 내용이 아니라, 유저가 **어떤 파일을 어디서 요청해야 하는가**를 모르는 상태로 시작한다는 점이다:

- `file_not_found`("파일을 찾을 수 없습니다")는 경로 차단과 달리 워크스페이스 개념을 전혀 언급하지 않는다.
- 신규 유저는 자신의 워크스페이스가 어디인지, 그 안에 무엇이 있는지 모른다.
- LLM이 "파일이 없다"는 피드백을 받아도 유저에게 의미 있는 안내를 생성하기 어렵다 — 에이전트가 접근 가능한 경로 정보를 응답에 포함할 수 없기 때문이다.

핵심 마찰: **빈 워크스페이스 + 외부 파일 요청 = LLM이 수렴 루프에 빠지고 유저는 이유를 모른다.**

## 제안

두 가지 레이어에서 개선이 필요하다:

### 레이어 1 — `file_not_found` 메시지에 워크스페이스 컨텍스트 추가

현재: `"파일을 찾을 수 없습니다: {{path}}"`

제안: `"파일을 찾을 수 없습니다: {{path}} (에이전트 워크스페이스: {{workspace}})"`

유저가 `file_not_found`를 볼 때 자신의 워크스페이스 경로를 함께 보여주면, "아, 내 워크스페이스에 이 파일이 없구나"를 즉시 이해할 수 있다.

### 레이어 2 — 신규 유저 첫 진입 시 워크스페이스 안내 (선택적)

빈 워크스페이스 상태를 감지하면 StatusBar 또는 시스템 메시지로 한 줄 안내:
"에이전트 워크스페이스가 비어있습니다. 파일 작업은 {{workspace}} 안에서만 가능합니다."

이 제안은 구현 범위가 넓으므로, 레이어 1만 먼저 적용해도 충분한 개선 효과가 있다.

## 근거

신규 가입자는 워크스페이스 개념을 자연스럽게 이해하지 못한다. `file_not_found`만으로는 "파일이 존재하지 않음"과 "에이전트가 접근 불가능한 경로"를 구분할 수 없다. 워크스페이스 경로가 메시지에 포함되면 유저가 올바른 위치에 파일을 옮기거나, 접근 가능한 경로로 요청을 변경하는 구체적 행동을 취할 수 있다.
