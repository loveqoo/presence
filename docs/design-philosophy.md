# presence: 설계 철학

이 문서는 presence 를 왜 만드는지, 어떤 구조를 지향하는지, 어떤 원칙이 구체적 결정을 이끌어야 하는지를 적어둔다. 구현 세부사항은 코드와 스펙이 가진다. 이 문서는 그것들이 어느 방향을 향해야 하는지에 대한 북극성이다.

## 왜 만드는가

이미 좋은 도구들이 많다. 메이저 AI 회사들은 대부분의 표면 기능 — todo, 스케줄링, 다기기 동기화, A2A 연동 — 을 결국 더 잘 만들 것이다. 그럼에도 직접 만드는 이유는 두 가지다.

첫째, **이 프로젝트의 진짜 산출물은 코드가 아니라, AI 와 함께 일하는 법을 몸으로 배운 나 자신이다.** 메이저 도구를 *쓰는 것만으로는* 얻을 수 없는 종류의 지식이 있다. context rot 을 직접 겪고, 에이전트가 규칙을 어기는 걸 보고 hook 으로 경계를 만들고, 리팩토링 후 안 돌아가는 걸 발견하고 검증 체계를 고안하는 — 이런 경험에서만 얻는 지식. 이건 portable 하다. 다음 어떤 프로젝트에도 살아남는다.

둘째, **presence 는 내 도구다.** 메이저 회사들은 median 유저에 최적화한다. FP 모나드 위에서 사고하고, 에이전트를 책임 영역으로 분리하고, 코드와 문서를 같은 호흡으로 진화시키는 사용자를 1순위로 지원할 도구는 거의 없을 것이다. 자기에게 정확히 맞는 도구를 만들고 그 안에서 사는 경험은 대량생산 도구로 대체 불가능하다.

## 핵심 베팅

presence 는 **Free Monad 위에 Op ADT 를 얹고, 그 ADT 를 LLM 의 선택 공간으로 사용한다.** 이것이 내 핵심 베팅이다.

근거는 이렇다. LLM 이 자유 텍스트를 뱉어내도록 두면 그 출력이 결국 어떤 AST 로 수렴하는 과정이 암묵적이고 예측 불가능해진다. AST 를 먼저 정의하고 LLM 이 그 안에서 *선택하게* 만들면, LLM 의 자유도가 finite 한 공간으로 제한되고 출력 variance 가 Op 개수만큼으로 bounded 된다. **LLM 은 발명하지 않고 선택한다.** 선택이 발명보다 결정적으로 검증 가능하다.

이 베팅이 틀릴 수도 있다. 산업은 imperative / graph-based 방향으로 표를 던졌고, 그들은 빠르다. 그러나 "산업이 안 한다" 는 약한 증거다. 산업은 빠른 시장 진입에 최적화된 사람들로 구성되어 있고, 새로운 substrate 시도는 그 반대 방향이다. "검증된 적 없음" 과 "안 통함" 은 다른 말이다. 빈 공간에서 답을 찾는 건 무모함이 아니라 연구다.

그리고 베팅이 틀렸다고 판명되더라도 잃는 건 코드뿐이고, 얻는 건 "왜 안 되는지에 대한 first-hand 지식" 이다. 이기든 지든 학습은 비대칭적으로 가치 있다.

## 구조 3층: 서로 간섭하지 않는 가독성

presence 는 세 계층으로 구성된다. 각 계층은 다른 두 계층을 몰라도 읽을 수 있어야 한다.

1. **Op 정의 계층** — 각 Op 는 잘 구성된 인터페이스 위의 템플릿 메소드다. 도메인 불변식(validation, policy, audit, sanitize) 이 템플릿의 뼈대에 고정되고, 변하는 부분만 injected dependency 로 열린다. 이 계층을 읽으면 *도메인 알고리즘* 을 이해할 수 있다. 하위 구현을 볼 필요 없다. *(OOP 다형성의 고전적 이점)*

2. **AST 합성 계층** — Op 들을 Free Monad 로 조립한 프로그램이다. 이 계층을 읽으면 *시스템이 무엇을 하려는지* 의 sequence 를 이해할 수 있다. 인터프리터를 볼 필요 없다. *(Free Monad 의 고전적 이점)*

3. **Runner / Interpreter 계층** — 합성된 AST 를 실제로 실행한다. mock, prod, delegate 등 여러 interpreter 가 존재한다. 이 계층을 읽으면 *어떤 환경에서 어떻게 실행되는지* 를 이해할 수 있다. 도메인 로직을 다시 읽을 필요 없다. *(layered architecture 의 고전적 이점)*

이 3층 분리가 무너지지 않는 한, presence 는 국소 변경에 강건하다. 한 계층을 바꿔도 다른 계층은 다시 읽지 않아도 된다. 이것이 큰 시스템에서 폭발적인 가치를 가진다.

## Op ADT 는 도메인 어휘다

presence 의 Op ADT 는 *원시 연산의 모음* 이 아니다. *도메인 어휘* 다.

원시 연산(`ReadFile`, `WriteFile`, `HttpGet` 같은 것) 의 ADT 는 조립의 부담을 위쪽으로 전가한다. 사용자(또는 LLM) 가 매번 벽돌을 올바르게 쌓아야 한다. 불변식은 각 조립 지점에서 재확인되어야 한다.

presence 의 Op 는 그 대신 도메인 레벨의 완성된 연산이다 — `PersistUserMemory`, `RecallRelevantContext`, `DelegateToSubAgent` 같은. 각 Op 내부에는 이미 validation, policy check, audit log, 필요한 경우 retry 와 rollback 까지 template method 의 뼈대로 녹아 있다. **이 Op 를 선택하는 쪽은 내부 규율을 걱정할 필요가 없다.** 인터프리터 역시 각 Op 의 실행 전략만 바꿀 뿐 내부 불변식을 재조립하지 않는다.

이것이 중요한 이유는 LLM 이 선택하는 단위가 올라가기 때문이다. 원시 연산을 조립하게 만들면 LLM 은 매번 올바른 조립을 재발명해야 하지만, 도메인 어휘에서 선택하게 만들면 *이미 올바른 연산들 중 무엇이 적절한가* 만 결정하면 된다. 선택 공간은 좁아지고 선택의 결과는 예측 가능하다.

trade-off 는 있다. Op 개수가 늘고 *합성의 표현력* 이 약간 떨어진다 — 각 Op 가 이미 충분히 완성되어 있어서 여러 개를 조합해야 할 이유가 줄어든다. 대신 *단일 Op 의 안전성* 과 *선택의 명료함* 을 얻는다. agent 시스템에서는 이게 좋은 거래다. agent 가 추론하는 단위가 애초에 "유저 의도 기억", "관련 맥락 조회", "외부 도구 호출" 같은 도메인 레벨이기 때문이다. Op ADT 를 그 레벨에 맞추는 게 agent 의 자연스러운 추상화 높이와 일치한다.

## 합성 가능한 부품 — algebraic law 로서의 contract

presence 의 주요 구성 요소 — 메모리, UserContext, sub-agent, 도구 — 는 모두 합성 가능해야 한다.

단순히 "interface 만 맞추면 교체 가능" 이라는 뜻이 아니다. **합성 가능성이 algebraic law 수준에서 보장되어야 한다.** 두 메모리 backend 를 합쳤을 때 결과가 어떤 semigroup 을 형성해야 하고, 두 UserContext 를 combine 했을 때 각 필드가 자기만의 combine 의미(union, priority-prepend, fallback chain 등) 를 갖는 product semigroup 이어야 하고, sub-agent 를 조합하는 연산자는 sequential / parallel-with-vote / hierarchical / fallback 중 어느 의미인지 명시되어야 한다.

이것을 JavaScript 에서 어떻게 강제할 것인가. 답은: **contract 를 "법칙 테스트 묶음(law spec)" 으로 정의해서 배포하고, 어떤 implementation 이든 이 spec 을 통과해야만 plug 가능하게 만든다.** 어차피 Haskell 의 type system 조차 결합법칙을 컴파일 타임에 enforce 하지 못한다. 모든 언어에서 law 는 결국 test suite 와 문서화된 contract 로 검증된다. JavaScript 는 signature 단계의 sanity check 를 잃을 뿐이고, 그건 property-based testing 으로 상당 부분 메울 수 있다. fp-ts 나 effect-ts 가 이미 이 길을 걷고 있다.

왜 이게 중요한가. 나중에 mem0 가 더 좋은 메모리 구현으로 교체되더라도, 새 구현이 같은 law spec 을 통과한다면 합성 레벨에서의 코드 변경은 0 이다. **개별 구현의 진화가 합성 구조를 흔들지 않는다.** 이것이 presence 가 시간을 견디는 방식이다.

## 두 레벨의 pluggability 를 구분하라

presence 는 두 종류의 swap 지점을 계층적으로 겹친다.

1. **Op 내부의 DI (템플릿 메소드 레벨)** — Op 가 받는 interface 에 구체 구현을 주입한다. `PersistUserMemory` 가 `Memory` 인터페이스를 받고 mem0 든 로컬 JSON 이든 주입할 수 있다. Op 는 *실제로* 작동하지만 백엔드만 바뀐다. **도메인 의존성** 이 이 레벨에 속한다 — Memory, LLM, Tool 실행기, 저장소, 외부 API. Op 의 본질적 협력자이고 구현이 바뀌어도 Op 의 의미는 동일하다.

2. **Runner (인터프리터) 레벨 swap** — Op ADT 전체를 다른 interpreter 가 해석한다. mock runner 는 Op 를 실행하지 않고 기록만, prod runner 는 실제 실행, delegate runner 는 다른 서버로 위임. **횡단 관심사** 가 이 레벨에 속한다 — 감사 로그, trace, logging, rate limit, retry. Op 의 본질이 아니라 *실행의 측면* 이다. 이걸 각 Op 에 DI 로 넣기 시작하면 Op 마다 같은 의존성이 여러 개 따라다녀서 도메인 신호가 묻힌다.

이 둘은 "같은 개념의 두 얼굴" 이 아니라 *서로 다른 차원* 이다. 같이 썼을 때 비로소 그림이 완성된다. Scala 의 ZIO, TypeScript 의 Effect-TS 가 독립적으로 같은 결론에 도달했다 — Free + DI 의 계층 구성이 우연이 아니라 *이 설계 공간의 자연스러운 균형점* 이라는 증거다.

## 수렴하는 시스템: 결정론은 목표가 아니다

presence 는 *처음부터 결정론적일 필요가 없다.* 수렴해가는 시스템이면 된다.

LLM 의 비결정성은 substrate 의 객관적 속성이 아니라 *우리의 모델링 해상도와 substrate 사이의 간극* 이다. 모델링이 정교해질수록 같은 substrate 가 "부수효과" 에서 "알려진 분포를 가진 연산" 으로 이동한다. **예상할 수 있다면 더 이상 부수효과가 아니다.** IO 도 같은 여정을 걸었다 — 한때는 두려움의 대상이었다가 점차 통제 가능한 effect 로 옮겨왔다. LLM 은 같은 여정의 훨씬 이른 단계에 있을 뿐이다.

중요한 건 *수렴을 가능하게 하는 구조* 다. 그 구조는 대략 이렇게 생겼다:

- **외부 환경** = LLM 호출 (high-variance substrate)
- **내부 상태** = 메모리, 세션 상태, UserContext (환경 모델의 축적)
- **경계면** = Op ADT, 인터프리터, prompt 구성 (환경과의 상호작용을 매개하는 finite 한 경계)
- **adaptive loop** = 반복 사용 + 경험 축적 + 구조 갱신

이 배치 안에서 내부 상태는 환경의 패턴을 암묵적으로 학습하고, 반복될수록 "예측" 과 "관찰" 의 간극이 좁아진다. 결정론적 수렴이 아니라 *분포의 수렴* 이지만, 충분히 안정된 분포는 실용적으로 결정론과 구분되지 않는다. 작은 변화를 무한히 반복하면 자연상수 같은 안정점이 드러나듯이.

presence 의 구성 요소 하나하나 — 추상화된 메모리, 합성 가능한 UserContext, 합성 가능한 sub-agent, Op ADT 를 통한 finite 선택 공간 — 은 이 수렴 루프의 부품이다. "깔끔한 설계" 가 아니라 *수렴을 가능하게 만드는 구조적 요건* 이다.

### 미묘한 제약: model drift

수렴은 *환경이 정적이거나 천천히 변할 때* 성립한다. 그런데 LLM 은 내부적으로 빠르게 변한다 — 새 버전, fine-tuning, prompt sensitivity drift. 그래서 수렴이 *특정 model version 안에서만* 성립하고, 모델이 바뀌면 일정 부분 재수렴이 필요하다.

대비는 구조적으로 가능하다. 모델 독립적인 부분(Op ADT, memory schema, UserContext 구조) 과 모델 의존적인 부분(prompt 구성, 해석 휴리스틱) 을 분리해두면, 모델 전환 시 재학습 비용이 후자에만 국한된다. presence 의 인터프리터 분리가 이미 이 방향으로 가고 있다.

## Agent Adaptation Memory: 자기 자신에 대한 내부 상태

presence 의 내부 상태는 두 층으로 구성된다. 하나는 *외부 세계에 대한 모델* — 유저 선호, 과거 결정, 맥락 — 으로 지금까지 이야기한 user memory 가 맡는다. 다른 하나는 *내부 에이전트 자신에 대한 모델* — 이 시스템이 LLM 과 함께 반복적으로 드러내는 선택 오류와 편향. 이걸 **Agent Adaptation Memory** 라 부른다.

LLM 은 상태를 갖지 않는 고분산 substrate 다. Adaptation Memory 는 그 비결정성을 내부 상태로 끌어들여, 오차를 관찰 가능하고 재사용 가능한 형태로 만든다. 에이전트가 자기 자신을 학습하는 층이다.

이 메모리는 세 가지를 기록한다:

- **오해 패턴** — 특정 Op 를 다른 Op 로 혼동하거나, 선택 편향이 반복되는 경우
- **실패 이력** — 어떤 조건에서 어떤 선택이 실패했는지
- **교정 효과** — 어떤 피드백이 실제로 선택을 개선했는지

retry 턴에서 이 메모리는 단순한 실패 사유를 넘어서, 다음 선택을 제한하거나 유도하는 soft constraint 로 사용된다. **이 메모리는 규칙이 아니라 bias 다.** 선택을 금지하지 않고, 확률 분포를 미세하게 이동시킨다. 규칙은 brittle 하지만 bias 는 recover 여지를 남긴다. 비결정성을 제거하려 애쓰지 말고 bounded 분포로 만들라는 원칙과 직접 맞물린다.

주의할 점:

- **model/version scoped** — 모델이 바뀌면 오차 특성도 바뀐다. 이 memory 는 바로 위에서 이야기한 model drift 와 짝을 이룬다.
- **과보정 방지** — 특정 선택을 과도하게 회피하지 않도록 제한. bias 메커니즘은 반드시 decay 와 반대 증거 수용을 내장해야 한다.
- **유저 메모리와 분리** — 외부 세계 모델과 내부 오차 모델은 다른 계층이다. retrieval 패턴부터 다르다 (user memory 는 의미 유사도로, adaptation memory 는 현재 Op 선택 맥락으로).

Adaptation Memory 는 presence 의 수렴 구조를 가속하는 부품이다. 실패는 제거되지 않는다. 대신 더 빠르게, 더 좁은 분포로 수렴한다.

### 선택의 evidence: Op 에 attached statistics

Adaptation Memory 는 단순히 과거 오차의 창고가 아니라, **Op 정의와 함께 empirical evidence 를 제시하는 방식** 으로 선택에 개입한다.

각 Op 는 prompt 제시 시점에 자신의 empirical 이력을 동반한다. 성공률, 직전 Op 와의 transition 성공률, 현재 context 에서의 조건부 성공률. LLM 은 이 evidence 를 받아 prior(학습된 지식) + context(현재 상황) + posterior(이 instance 의 실측) 세 층을 합성해 선택한다.

**통계는 prior 를 대체하지 않는다. posterior 를 갱신할 뿐이다.** 이것이 bias 와 rule 의 차이다. 과거에 실패한 선택을 금지하지 않고, 같은 선택이 다른 맥락에서 여전히 옳을 수 있는 여지를 남긴다.

특히 **이전 Op 와의 transition 통계는 Op ADT 위에 learned grammar 를 얹는 것과 같다.** Op 가 도메인 *어휘* 라면, 통계가 축적하는 건 그 어휘의 *문법* 이다. 이 문법은 LLM 의 training 에 없는 정보 — presence 고유의 local knowledge 다. Op 를 도메인 레벨로 구체적으로 정의했기 때문에 자연스럽게 얻을 수 있는 이점이다. 원시 연산으로 짠 ADT 위에는 이런 문법이 형성되지 않는다 — 매 조합마다 유효성이 달라지기 때문이다.

구현 세부(aggregation, confidence, exploration 정책) 는 구현 시점에 결정하되, 원칙은 하나다: **통계는 exploit 을 가속할 뿐 explore 를 닫지 않는다.**

## 지키려는 것 — 북극성

구현이 vision 에서 표류할 때 돌아올 기준점이다.

1. **Op 는 원시 연산이 아니라 도메인 어휘다.** Op 를 추가할 때 "이게 도메인 어휘인가, 아니면 원시 연산의 레벨에서 내가 편하려고 정의한 것인가" 를 먼저 물어라. 후자라면 이미 Op 정의가 틀렸다.

2. **Op 내부에는 template method 의 뼈대가 있어야 한다.** 외부에서 Op 를 사용하는 쪽이 validation, audit, policy 를 재조립해야 한다면 Op 가 제 일을 안 하고 있는 것이다.

3. **Op ADT / AST / Runner 3층은 서로 간섭하지 않아야 한다.** 한 계층의 변경이 다른 두 계층의 가독성을 흔들면, 계층 경계가 새고 있다는 신호다.

4. **plug 지점은 법칙 test suite 를 동반해야 한다.** 단순 interface 로 족하다고 판단되는 순간, algebraic 한 합성 가능성은 이미 포기된 것이다.

5. **DI 는 Op 내부에, 횡단 관심사는 Runner 로.** 이 둘을 섞으면 도메인 신호가 묻힌다.

6. **결정론은 목표가 아니라 한 특수한 경우의 수렴이다.** 비결정성을 제거하려 애쓰기보다, 비결정성을 bounded 분포로 만드는 구조를 설계하라.

7. **실패와 evidence 는 내부 상태의 입력이다.** 실패를 제거하려 하지 말고 관찰 가능한 형태로 흘려서 수렴 루프의 신호로 삼아라. Op 는 자신의 empirical 이력과 함께 제시되어야 한다.

8. **이 프로젝트의 진짜 산출물은 나 자신이다.** 코드가 언젠가 버려지더라도 배움이 남으면 프로젝트는 성공한 것이다.

## 이 문서의 성격

이 문서는 규칙이 아니라 *의도의 표명* 이다. 규칙은 `.claude/rules/` 에 있다. 스펙은 `docs/specs/` 에 있다. 운영 규약은 `CLAUDE.md` 에 있다. 이 문서는 그 셋이 왜 그 모양이어야 하는지에 대한 답이다.

vision 은 머릿속에만 있으면 증발한다. 한 달 후 구현에 파묻혀 방향을 잃을 때, 이 문서로 돌아와야 한다. 이 문서가 틀렸다고 느껴지는 순간이 오면, 문서를 바로 갱신하라. **코드가 문서를 서서히 추월하는 것보다, 문서가 코드를 의도적으로 끌어당기는 게 건강하다.**
